## Results Snapshot

### `pulith-fetch` `multi_source`

- priority success-first: ~20.9-26.0 ms across 2/4/8 candidates
- priority fallback: ~50.1 ms (3), ~98.5 ms (5), ~157.3 ms (8)
- race one-success: ~27.6-32.4 ms across 3/5/8 candidates

Interpretation:

- fallback behavior scales roughly with attempted candidate count as expected
- race strategy remains lower-latency under partial failures than ordered fallback

### `pulith-install` `pipeline`

- fetch->store->install: ~65.5 ms (1 MiB), ~71.8 ms (8 MiB)
- archive extract->store->install: ~34.2 ms (1 MiB), ~33.9 ms (8 MiB)

Interpretation:

- pipeline timings remain in expected range; no threshold change is indicated by this run alone

### `pulith-install` `copy_transition`

Selected medians (time):

- 64 KiB: `adaptive_4m` ~6.64 ms (best among tested variants)
- 256 KiB: `adaptive_4m` ~7.89 ms (best among tested variants)
- 1 MiB: `adaptive_8m` ~9.34 ms (slightly better than `adaptive_4m` ~9.92 ms)
- 4 MiB: `adaptive_4m` ~8.46 ms (best among tested variants)
- 8 MiB: `adaptive_4m` ~9.00 ms (best among tested variants)
- 16 MiB: `adaptive_1m` ~9.71 ms (best among tested variants)

Interpretation:

- current default copy-only threshold (`4 MiB`) remains a reasonable cross-size compromise
- larger-size wins vary between `1 MiB` and `4 MiB` adaptive settings; a single machine run is not sufficient for threshold change

## Decision for Block G

- keep `DEFAULT_COPY_ONLY_THRESHOLD_BYTES = 4 * 1024 * 1024` unchanged
- treat this dataset as baseline evidence only; do not retune thresholds until repeated runs are captured on steadier CI-class machines
